‚óŹ Enough context. Analysis as lead eng ‚ÄĒ Signal at 100+ users.

  Scaling gaps, by severity

  ūüĒī Blockers (break or degrade fast)

  1. Reaction counts computed client-side by fetching every reaction row.
  decorateNotes (notes.ts:92) pulls all reactions rows for the 10 notes on
  screen, counts in JS. One popular note w/ 5k reactions = 5k rows over wire per
  feed load. No aggregate. Fix: DB-side count ‚ÄĒ either reaction_counts
  denormalized column (trigger on insert/delete) or a count view. At 100 users
  survivable; at viral note, not.

  2. Feed has no realtime ‚ÄĒ goes stale.
  useFeed loads once, needs manual pull-refresh. 100 users posting = feed always
  behind. No supabase.channel().on('postgres_changes'). Add realtime
  subscription on voice_notes insert ‚Üí prepend, and on reactions ‚Üí patch counts.

  3. Profile loads ALL notes unbounded.
  fetchUserNotes (notes.ts:237) = no limit, pulls every note + nested reactions.
  fetchProfileStats + fetchPublicProfile both call it just to .length + sum.
  Power user w/ 500 notes = huge payload every profile open. Fix: count:'exact',
  head:true for totals; paginate the note list.

  ūüü† Correctness / conflict risks

  4. Username signup race.
  users.username unique ‚ÄĒ two users pick same name same instant ‚Üí one hits
  23505, need graceful "taken, retry" not crash. Check UsernameScreen handles
  it.

  5. Migration drift ‚ÄĒ already bit you.
  0005_fix_reactions.sql exists ONLY because 0002 never reached prod DB
  (42501/42P10 in prod). Manual SQL-editor deploys = silent skip. This WILL
  recur w/ more devs/envs. Fix: supabase db push in CI, verify migration state,
  stop hand-pasting.

  6. "Following" feed in(user_id, authorIds).
  notes.ts:153 ‚ÄĒ inlines all followed ids into query. Follow 1k users ‚Üí giant IN
  list, URL/statement bloat. Fix: server-side join (RPC or view joining
  follows).

  ūüĒī Security / abuse (write normal ‚ÄĒ matters at 100 users)

  7. No block / mute / report. Global public feed of voice recordings + 100
  strangers = harassment vector with zero moderation primitives. Need: block
  table (filters feed), report flow, and an admin takedown path. This is the
  biggest social-scaling gap, not a perf one.

  8. Audio is a fully public bucket, permanent URLs. Anyone (even
  unauthenticated, with the URL) can fetch any voice note forever ‚ÄĒ URLs are
  guessable-ish (<uid>/<timestamp>.m4a) and leak the poster's uid. Consider
  signed URLs + authenticated read policy if notes are meant to be in-app only.

  9. No upload rate limit / size cap. A user can spam-post to inflate storage
  cost and flood the feed. Add: per-user post rate limit (edge function or DB
  trigger counting recent inserts) and a max file size / max duration enforced
  server-side, not just client.

  ūüü° Cleanup / operational

  10. Dead deliver-note edge function + deliveries table. Unused since
  global-feed migration (0002) but still deployed, still has RLS surface. Remove
  or document ‚ÄĒ dead auth surface is a liability.

  11. public_usernames view exposes every username to every authed user. Fine
  for this app, but note it ‚ÄĒ it's the enumeration surface (scrape all users).
  Acceptable, flag it.

  12. Feed audio players ‚ÄĒ memory. Many AudioPlayer mounted in list. Verify
  unmount/recycle on scroll (FlatList windowing) else 100-item scroll = memory
  bloat / audio handle leak.

  13. No offline/retry. Errors throw raw strings; no retry, no queue for posting
  on flaky mobile network.

  4. Block + report primitives ‚ÄĒ #7
  5. CI migrations (db push) ‚ÄĒ #5
